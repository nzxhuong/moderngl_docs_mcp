### Context Creation > Introduction

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/context.md

A context provides ModernGL with access to OpenGL instructions. The method of context creation varies by operating system and target platform. Typically, the default backend for your OS is automatically selected unless a specific 'backend' parameter is provided. Default backends include WGL on Windows, GLX on Linux, and CGL on OS X. ### An introduction to OpenGL

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/intro.md

OpenGL, created in 1992 by Silicon Graphics and now managed by the Khronos Group, is a standard API for hardware-accelerated rendering across multiple platforms. Vendors implement the specifications to provide this functionality.

### Moderngl Documentation > Context.query() — GPU Performance Counters

Source: https://context7.com/moderngl/moderngl/llms.txt

Context.query() wraps OpenGL queries for GPU performance monitoring. It can track the number of fragments that passed the depth test (`samples`), total primitives generated (`primitives`), and GPU time elapsed in nanoseconds (`elapsed`).

### Context Creation > Specifying context backend

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/context.md

### An introduction to OpenGL > The simplified story

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/intro.md

The "Fixed Pipeline" (Old OpenGL) used functions like glVertex and glBegin/glEnd, with strong opinions and limitations. In contrast, the "Programmable Pipeline" (Modern OpenGL) uses functions like glCreateProgram and glUniform, working with data buffers and GPU-run programs called shaders written in GLSL, offering greater flexibility.

--------------------------------

### Moderngl Documentation > Context.compute_shader() — General-Purpose GPU Computing

Source: https://context7.com/moderngl/moderngl/llms.txt

Compute shaders enable general-purpose GPU computing (GPGPU) tasks independent of the rasterization pipeline. Data is passed through storage buffers or image units. This feature requires OpenGL 4.3 or later.

--------------------------------

### An introduction to OpenGL > Where does ModernGL fit into all this?

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/intro.md

ModernGL is a Pythonic wrapper for OpenGL 3.3 core or higher, exposing the Programmable Pipeline through objects like Buffer and Program. It simplifies OpenGL by hiding complex details and removing common pitfalls, focusing on shaders and the rendering pipeline.

--------------------------------

### Using GLSL in ModernGL

Source: https://github.com/moderngl/moderngl/blob/main/README.md

GLSL (OpenGL Shading Language) is essential for shader programming in ModernGL, enabling GPU execution for enhanced graphics rendering. Key concepts include standard data types (int, float, double, bool) and graphics-specific types like vectors (vec2-vec4) and matrices (mat2-mat4).

--------------------------------

### Moderngl Documentation

Source: https://context7.com/moderngl/moderngl/llms.txt

Scopes in Moderngl provide a convenient way to manage rendering states, including framebuffers, pipeline states, textures, and buffers. They can be used as context managers for temporary state changes or assigned to VertexArrays to ensure specific states are active during rendering. Scopes can also be combined with queries for performance monitoring.

--------------------------------

### An introduction to OpenGL > The simplified story

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/intro.md

OpenGL has evolved into "Modern OpenGL", with version 3.3 being the first to modernize the API, enabling utilization of Direct3D 11-class hardware. This version and subsequent 4.x versions are forward compatible. Running in "core mode" removes old features, while "compatibility mode" allows mixing old and new APIs.

--------------------------------

### Moderngl Documentation > Context.enable() / Context.enable_only() / Context.disable() — Pipeline State Flags

Source: https://context7.com/moderngl/moderngl/llms.txt

Pipeline state flags like depth testing, face culling, and blending are controlled using `enable()`, `enable_only()`, and `disable()` methods. `enable_only()` atomically replaces all current flags, while `enable()` and `disable()` modify individual flags. Advanced configurations for blending and face culling are supported, along with direct enabling of OpenGL capabilities.

### Buffer Format

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/buffer_format.md

When rendering multiple instances of objects, different buffers can have distinct usage specifiers. For example, `vbo_verts_normals` can contain vertices and normals for a single object, `vbo_offset_orientation` can store positioning and orientation data for each instance, and `vbo_colors` can hold color information. The usage specifier determines how data from these buffers is applied: `/v` applies data per vertex, `/i` applies data per instance, and `/r` applies data once per render call to all vertices and instances.

--------------------------------

### Buffer Format

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/buffer_format.md

The `/i` usage specifier indicates that data is passed to all vertices within an instance, and then the next value in the buffer is used for the next instance. This is suitable for per-instance data such as offsets or transformations.

### Summary

Source: https://context7.com/moderngl/moderngl/llms.txt

Integration with ModernGL follows a lifecycle of creating persistent objects at startup, updating uniforms and buffers per frame, rendering using VAOs or compute shaders, and reading results from framebuffers. State management is best handled with Scope objects to ensure consistent settings. For thread-safe applications or contexts managed by other libraries, set `ctx.gc_mode = 'context_gc'` and call `ctx.gc()` per frame for safe memory reclamation. The Query object aids in GPU performance profiling by providing timing and sample counts within the render loop.

### Buffer Format

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/buffer_format.md

A buffer format is a short string describing the layout of data in a vertex buffer object (VBO). It specifies what each element of the array looks like, for example, a buffer containing an array of high-precision 2D vertex positions might have the format "2f8", meaning each element consists of two floats, each 8 bytes wide (doubles). Buffer formats are used in the Context.vertex_array() constructor.

### Context Creation > Context Sharing

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/context.md

Object sharing between contexts is an experimental feature that allows ModernGL objects to be used across multiple contexts if the underlying OpenGL context supports it and the 'share' parameter is enabled during creation. Objects have a 'ctx' property indicating their creation context, but ModernGL does not enforce context checking for shared objects. This feature has limitations, particularly with objects that reference other objects like framebuffers or vertex array objects.

### Blend Equations

Source: https://github.com/moderngl/moderngl/blob/main/docs/reference/context.md

Blend equations define how source and destination colors are combined. Context.FUNC_ADD performs source + destination. Context.FUNC_SUBTRACT performs source - destination. Context.FUNC_REVERSE_SUBTRACT performs destination - source. Context.MIN selects the minimum of the source and destination colors, while Context.MAX selects the maximum.

### Context.framebuffer() — Off-Screen Render Target

### Context.gc_mode

### Program

Source: https://github.com/moderngl/moderngl/blob/main/docs/reference/program.md

A Program object represents fully processed executable code in the OpenGL Shading Language, for one or more Shader stages. In ModernGL, a Program object can be assigned to VertexArray objects, which bind the Program when render() or transform() is called. Program objects do not have a use() method; VertexArrays encapsulate this mechanism. Programs cannot be instantiated directly and require a context, obtained via Context.program().

### Buffer

Source: https://github.com/moderngl/moderngl/blob/main/docs/reference/buffer.md

Buffer objects are OpenGL objects that store an array of unformatted memory allocated by the OpenGL context, essentially data allocated on the GPU. They are used to store vertex data, pixel data retrieved from images or the framebuffer, and various other types of information. Buffer objects cannot be instantiated directly and require a context; creation is done using `Context.buffer()`. Content can be copied using `Context.copy_buffer()`.

### ModernGL Types

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/getting_started/low_start/moderngl_types.md

ModernGL utilizes several key types for graphics operations. A `Buffer` stores vertex data in graphics memory. A `Program` represents a shader program, configured with GLSL source code. A `VertexArray` facilitates communication between `Buffer` and `Program` for rendering calls and is immutable but cheap to create. Various `Texture` types exist, with `Texture` being the most common 2D texture. A `Framebuffer` serves as an offscreen render target, supporting attachments like textures and depth buffers.

### Standalone rendering

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/getting_started/first_rendering/rendering.md

Standalone rendering in ModernGL enables rendering without the need for a window. Rendering is initiated when the `VertexArray.render()` method is invoked. The default rendering mode is `moderngl.TRIANGLES`, but this can be changed to other modes like `moderngl.LINES` or `moderngl.LINE_STRIP` as needed. To display the rendering output, the Pillow (PIL) library can be used to transfer the texture from GPU memory to RAM and then display it.

### Sampler

Source: https://github.com/moderngl/moderngl/blob/main/docs/reference/sampler.md

A Sampler Object stores sampling parameters for texture access within a shader. ### Uniform

### Texture Formats > Normalized Integer Textures

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/texture_formats.md

Normalized integer textures treat integer texel values as normalized floating-point numbers in shaders, typically in the range [0.0, 1.0]. Unsigned normalized textures (nu1, nu2) use GL_UNSIGNED_BYTE and GL_UNSIGNED_SHORT respectively, mapping their full integer range to [0.0, 1.0]. There is no standard for 32-bit normalized integer textures due to precision limitations.

### Context.clear_samplers()

Source: https://github.com/moderngl/moderngl/blob/main/docs/reference/context.md

The `clear_samplers()` method unbinds samplers from texture units. While sampler bindings clear automatically between frames, this method offers a way to ensure texture units are clear, preventing potential bugs. It accepts `start` and `stop` parameters to define the range of texture units to clear.

### Polygon offset (avoid z-fighting when drawing wireframe over solid)

Source: https://context7.com/moderngl/moderngl/llms.txt

Polygon offset can be used to push geometry away from the camera, which is useful for avoiding z-fighting when drawing wireframes over solid objects. Set `ctx.polygon_offset` to `1.0, 1.0` to enable it, and `0.0, 0.0` to disable it. Wireframe mode for debugging can be enabled by setting `ctx.wireframe` to `True`.

### Program

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/getting_started/first_rendering/program.md

Screen coordinates in ModernGL range from -1 to 1 for both x and y axes, with the point (-1, -1) representing the lower-left corner of the screen. A sample program can pass input vertex coordinates directly to these screen coordinates.

### The Lifecycle of a ModernGL Object > Manually Releasing Objects

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/gc.md

When `gc_mode` is set to `None`, objects in ModernGL do not automatically release their OpenGL resources. Each object type has a `release()` method that must be explicitly called to clean up all associated OpenGL resources.

### The Lifecycle of a ModernGL Object > Detecting Released Objects

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/gc.md

To detect if a ModernGL resource has been released, you can check the type of its internal object via the `.mglo` property. After calling `release()`, the type will change to `_moderngl.InvalidObject`.

### Buffer

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/getting_started/first_rendering/buffer.md

The size of a buffer can be changed at any time using the `Buffer.orphan(size=...)` function. After resizing, data can be written to the buffer using `Buffer.write(data)`, which transfers data from RAM to GPU memory. If the buffer size changes after being added to a VertexArray, the new number of vertices must be specified in the `vertices` parameter of `VertexArray.render()` or `VertexArray.transform()`.

### Other Enums

Source: https://github.com/moderngl/moderngl/blob/main/docs/reference/context.md

Barrier bits are used to control memory synchronization in ModernGL. These include bits for vertex attribute arrays, element arrays, uniforms, texture fetches, shader image access, command buffers, pixel buffers, texture updates, buffer updates, framebuffers, transform feedback, atomic counters, shader storage, and a bit for all barrier types.

### Vertex Shader (transforms)

Source: https://github.com/moderngl/moderngl/blob/main/docs/the_guide/getting_started/low_start/shader_transform.md

A known bug in some OpenGL drivers affects the `mod(number, divisor)` function. When the first argument is exactly twice the second, the function may return the divisor value instead of 0. To circumvent this, a custom `mod2` function can be implemented and used instead of the standard `mod` function.

### The Lifecycle of a ModernGL Object > The Threading Issue

Source: https://github.com/moderngl/moderngl/blob/main/docs/topics/gc.md

When using threads in Python applications, the garbage collector can run in any thread, which conflicts with OpenGL's requirement that only the main thread interacts with the context. In such scenarios, use the `context_gc` mode and periodically call `Context.gc()`, for instance, during every frame swap.
